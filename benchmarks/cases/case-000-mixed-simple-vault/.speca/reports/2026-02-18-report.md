## Threat Model

### Actors

| Actor | Trust Level |
|-------|------------|
| contract_owner | TRUSTED |
| whitelisted_depositor | SEMI_TRUSTED |
| external_caller | UNTRUSTED |

### Trust Boundaries

- contract_owner -> all functions including admin (withdraw, pause, unpause, whitelist management)
- whitelisted_depositor -> deposit() only
- external_caller -> no authorized access (should be rejected)

### Assumptions

- EVM execution is deterministic
- Block timestamp can be manipulated within ~15 seconds
- Solidity compiler version is ^0.8.20 (built-in overflow protection)
- msg.sender cannot be spoofed within the EVM
- External calls via .call{} may trigger reentrant execution

## Findings

### [FIND-001] Missing whitelist/owner access control in deposit() (Critical)

**Severity:** Critical
**Checklist Item:** CHK-ACCESS-001-a
**Requirement:** R-AUTH-003
**False Positive Risk:** Low

**Description:**
The deposit() function at line 23 is callable by any external address. The specification R-AUTH-003 requires that only whitelisted addresses or the owner may call deposit(). The function has no require() check, modifier, or any other mechanism to enforce this access restriction. This violates the authorization model of the contract: any UNTRUSTED external_caller can deposit ETH into the vault, bypassing the whitelist entirely. Combined with FIND-002 (anyone can unpause), an attacker can interact with the vault in ways the specification explicitly forbids.

**Proof Trace:**
```solidity
// ./examples/simple-vault/contracts/Vault.sol:23-28
    function deposit() external payable {
        require(!paused, "Paused");
        require(msg.value > 0, "Zero deposit");
        balances[msg.sender] += msg.value;
        emit Deposited(msg.sender, msg.value);
    }
```

**Reasoning:**
Specification says: R-AUTH-003 requires 'Only whitelisted addresses or the owner MUST be able to call deposit().' The interface.yaml also specifies access: whitelist_or_owner. Code does: deposit() at lines 23-28 only checks require(!paused) and require(msg.value > 0). There is no check against the whitelist mapping or owner address. Gap: Any address can call deposit() regardless of whitelist status. The whitelist state variable exists (line 7) and is managed by addToWhitelist/removeFromWhitelist, but deposit() never reads it. Impact: The entire whitelist authorization model is bypassed. Unauthorized actors can deposit ETH and have their balances credited, violating the contract's access control design.

**Recommendation:**
Add a whitelist/owner check at the beginning of deposit(). Either create a modifier or add an inline require:

```solidity
require(whitelist[msg.sender] || msg.sender == owner, "Not authorized");
```

This should be placed after the pause check and before the value check.

### [FIND-002] Missing onlyOwner modifier on emergencyUnpause() (Critical)

**Severity:** Critical
**Checklist Item:** CHK-ACCESS-001-b
**Requirement:** R-EMR-002
**False Positive Risk:** Low

**Description:**
The emergencyUnpause() function at line 45 lacks the onlyOwner modifier, allowing any external address to unpause the contract. The specification R-EMR-002 requires that only the owner may call both emergencyPause() and emergencyUnpause(). While emergencyPause() correctly has the onlyOwner modifier (line 40), emergencyUnpause() does not. This completely undermines the emergency pause mechanism: if the owner pauses the contract to prevent exploitation of another vulnerability, any attacker can immediately unpause it, re-enabling deposits and withdrawals.

**Proof Trace:**
```solidity
// ./examples/simple-vault/contracts/Vault.sol:45-47
    function emergencyUnpause() external {
        paused = false;
    }

// ./examples/simple-vault/contracts/Vault.sol:40-42 (comparison: emergencyPause has onlyOwner)
    function emergencyPause() external onlyOwner {
        paused = true;
    }
```

**Reasoning:**
Specification says: R-EMR-002 requires 'Only the owner MUST be able to call emergencyPause() and emergencyUnpause().' Code does: emergencyPause() at line 40 has the onlyOwner modifier (correct), but emergencyUnpause() at line 45 has no modifier — its signature is 'function emergencyUnpause() external' without onlyOwner. Gap: Any address can call emergencyUnpause() to set paused = false, negating the owner's emergency pause action. Impact: The emergency mechanism is rendered useless. An attacker who detects a pause can immediately unpause the contract. This is especially dangerous in combination with FIND-001: an attacker can unpause, then exploit the missing whitelist check to deposit and interact with the vulnerable contract.

**Recommendation:**
Add the onlyOwner modifier to emergencyUnpause():

```solidity
function emergencyUnpause() external onlyOwner {
    paused = false;
}
```

### [FIND-003] Reentrancy vulnerability in withdraw() — CEI pattern violated (High)

**Severity:** High
**Checklist Item:** CHK-REENT-001-a
**Requirement:** R-WDR-001, R-WDR-004
**False Positive Risk:** Medium

**Description:**
The withdraw() function at line 31 makes an external call via .call{value: amount} at line 35 without performing any state variable updates beforehand. There is no nonReentrant guard. The checks-effects-interactions (CEI) pattern is violated: the function checks (require statements at lines 32-33), emits an event (line 34), then immediately performs the external interaction (line 35) without any effects (state updates). Additionally, the balances mapping is never decremented during withdrawal, meaning there is no state to update — the accounting only tracks deposits, not withdrawals. If the owner address is a smart contract, its receive() or fallback() function can re-enter withdraw() before the first call completes. Since the balance check uses address(this).balance (which decreases with each ETH transfer), recursive calls would eventually exhaust the contract balance. This also relates to checklist items CHK-REENT-001-b (R-WDR-004: SHOULD follow CEI pattern) and CHK-REENT-002-a (cross-function reentrancy).

**Proof Trace:**
```solidity
// ./examples/simple-vault/contracts/Vault.sol:31-38
    function withdraw(uint256 amount) external onlyOwner {
        require(!paused, "Paused");
        require(amount <= address(this).balance, "Insufficient balance");
        emit Withdrawn(msg.sender, amount);
        (bool success, ) = msg.sender.call{value: amount}("");
        require(success, "Transfer failed");
        // state update after external call - reentrancy risk
    }
```

**Reasoning:**
Specification says: R-WDR-004 states 'Withdrawal SHOULD follow the checks-effects-interactions pattern.' R-WDR-001 states withdraw must transfer the specified amount to the owner. Code does: withdraw() at lines 31-38 performs checks (lines 32-33), then emits an event (line 34), then makes an external call (line 35). No state variable is updated at any point — neither before nor after the external call. The balances mapping is not decremented. Gap: The function violates CEI by performing an external call with no preceding state updates and no reentrancy guard. The comment at line 37 acknowledges this risk. Impact: If the owner is a smart contract (e.g., a multisig, DAO, or smart contract wallet), reentrancy through the receive() function is possible. Each reentrant call passes the balance check because address(this).balance decreases with each transfer. However, since withdraw() is restricted to onlyOwner (a TRUSTED actor), exploitation requires the owner's address to have reentrant callback logic. Severity is high (not critical) because the TRUSTED owner can already withdraw the full balance in a single call.

**Recommendation:**
1. Add a reentrancy guard (e.g., OpenZeppelin ReentrancyGuard with nonReentrant modifier) to withdraw().
2. If individual balance tracking is intended, decrement the appropriate balance BEFORE the external call.

```solidity
function withdraw(uint256 amount) external onlyOwner nonReentrant {
    require(!paused, "Paused");
    require(amount <= address(this).balance, "Insufficient balance");
    // effects before interactions
    emit Withdrawn(msg.sender, amount);
    (bool success, ) = msg.sender.call{value: amount}("");
    require(success, "Transfer failed");
}
```

3. At minimum, use the nonReentrant modifier as defense-in-depth even if the owner is trusted.
