# /speca-graph - Generate Program Graphs

You are generating program graphs from extracted specification requirements to formalize the logical structure of the specification. This is an optional enrichment step in the SPECA (SPEcification-to-Checklist Auditing) pipeline, inserted between `/speca-extract` and `/speca-map`. Program graphs improve downstream mapping precision and checklist quality by making implicit relationships between requirements explicit.

This step implements the V2 pipeline enhancement from the SPECA methodology (Figure 3: Spec → **Program Graphs** → Trust Model → Property → Checklist → Code → Audit Result).

---

## Phase 0: Prerequisites Check

1. Read `.speca/config.json` using the Read tool.
2. If the file does not exist, stop and tell the user: "No SPECA config found. Please run `/speca-init` first to initialize the project."
3. Extract the `language` field from the config. If `language` is not present, default to `"en"` (English). This setting determines the language for all human-readable text generated by this command (node labels, edge descriptions, console summary). Identifiers (requirement IDs, graph node IDs, relationship types) always remain in English.
4. Extract the `threat_model` object for boundary annotation.
5. Read `.speca/requirements.json` using the Read tool.
6. If the file does not exist, stop and tell the user: "No requirements found. Please run `/speca-extract` first to extract requirements from your specification files."
7. Parse the JSON and extract the `requirements` array. If the array is empty, stop and tell the user: "The requirements file contains no requirements. Please run `/speca-extract` to populate it."

---

## Phase 1: Identify Entities and Actions

Analyze each requirement to extract the fundamental entities and actions described in the specification.

### Step 1a: Extract Entities

For each requirement, identify mentioned entities:

- **Actors**: Subjects that perform actions (e.g., "owner", "user", "admin", "contract", "oracle", "msg.sender").
- **Assets**: Objects being acted upon (e.g., "token", "balance", "allowance", "vault", "fee", "ETH").
- **States**: Named contract states or modes (e.g., "paused", "initialized", "locked", "active").

For each entity, record:
- `entity_id`: A normalized identifier (e.g., `ACTOR-owner`, `ASSET-balance`, `STATE-paused`).
- `entity_type`: One of `actor`, `asset`, `state`.
- `label`: Human-readable name.
- `source_requirements`: Array of requirement IDs that reference this entity.

Deduplicate entities that refer to the same concept across requirements (e.g., "the owner" and "contract owner" are the same entity).

### Step 1b: Extract Actions

For each requirement, identify the actions (verbs) described:

- **State-changing actions**: deposit, withdraw, transfer, mint, burn, pause, upgrade, set, initialize.
- **Read actions**: query, check, verify, view, get.
- **Control flow actions**: revert, require, emit, call.

For each action, record:
- `action_id`: A normalized identifier (e.g., `ACTION-withdraw`, `ACTION-transfer`).
- `label`: Human-readable name (the verb form, e.g., "withdraw", "transfer").
- `action_type`: One of `state_change`, `read`, `control_flow`.
- `source_requirements`: Array of requirement IDs that describe this action.

## Phase 2: Build Relationship Graph

Construct a directed graph where nodes are entities and actions, and edges represent relationships.

### Step 2a: Define Nodes

Create a node for each unique entity and action identified in Phase 1. Each node has:
- `id`: The entity_id or action_id.
- `type`: `actor`, `asset`, `state`, or `action`.
- `label`: Human-readable name.
- `source_requirements`: Array of requirement IDs.

### Step 2b: Define Edges

For each requirement, derive directed edges between nodes. Edge types:

| Edge Type | Meaning | Example |
|---|---|---|
| `performs` | Actor performs an action | `ACTOR-owner` → `ACTION-withdraw` |
| `targets` | Action operates on an asset | `ACTION-withdraw` → `ASSET-balance` |
| `modifies` | Action changes a state | `ACTION-pause` → `STATE-paused` |
| `requires_state` | Action requires a precondition state | `ACTION-withdraw` → `STATE-active` (must be active) |
| `guards` | A condition guards an action | `ACTOR-owner` → `ACTION-setFee` (onlyOwner) |
| `emits` | Action triggers an event | `ACTION-deposit` → `EVENT-Deposit` |
| `depends_on` | One action depends on another | `ACTION-withdraw` → `ACTION-checkBalance` |

For each edge, record:
- `source`: Source node ID.
- `target`: Target node ID.
- `edge_type`: One of the types above.
- `requirement_id`: The requirement that defines this relationship.
- `constraint`: Optional constraint text (e.g., "msg.sender == owner", "amount > 0", "state != paused").

### Step 2c: Annotate Trust Boundaries

Using the `threat_model` from config.json, annotate edges with trust boundary information:

1. For each `performs` edge where the source is an actor, look up the actor's trust level in `threat_model.actors`.
2. For each `guards` edge, check if the guard corresponds to a trust boundary in `threat_model.boundaries`.
3. Add a `trust_annotation` field to relevant edges:
   - `trust_level`: The actor's trust level (`TRUSTED`, `SEMI_TRUSTED`, `UNTRUSTED`), or `null` if not in threat model.
   - `crosses_boundary`: Boolean indicating whether this edge crosses a defined trust boundary.

---

## Phase 3: Detect Structural Properties

Analyze the graph to detect structural properties that inform downstream analysis.

### Step 3a: Critical Paths

Identify paths from UNTRUSTED actors to high-value assets (assets referenced by requirements with `severity_hint: "high"`). These are the highest-risk attack surfaces.

For each critical path:
- `path`: Array of node IDs from actor to asset.
- `requirement_ids`: Array of requirement IDs along the path.
- `unguarded_steps`: Array of edges in the path that lack a `guards` edge (potential missing access control).

### Step 3b: State Dependency Clusters

Identify groups of actions that share state dependencies (modify or require the same state nodes). These clusters indicate:
- Potential reentrancy surfaces (shared state with external calls).
- State ordering requirements (action A must happen before action B).

For each cluster:
- `cluster_id`: `CLUSTER-NNN` (sequential).
- `shared_state`: The state node ID(s) shared by the cluster.
- `actions`: Array of action node IDs in the cluster.
- `requirement_ids`: Array of requirement IDs involved.

### Step 3c: Orphan Detection

Identify:
- **Orphan actions**: Actions with no actor performing them (missing access specification).
- **Orphan assets**: Assets with no actions targeting them (potentially dead state).
- **Disconnected requirements**: Requirements whose entities do not connect to the main graph.

### Step 4a: Build the Output JSON

Construct the output object with this schema:

```json
{
  "generated_at": "<ISO 8601 timestamp>",
  "total_nodes": <integer>,
  "total_edges": <integer>,
  "nodes": [
    {
      "id": "ACTOR-owner",
      "type": "actor",
      "label": "owner",
      "source_requirements": ["SPEC-AUTH-001", "SPEC-AUTH-002"]
    }
  ],
  "edges": [
    {
      "source": "ACTOR-owner",
      "target": "ACTION-withdraw",
      "edge_type": "performs",
      "requirement_id": "SPEC-AUTH-001",
      "constraint": "msg.sender == owner",
      "trust_annotation": {
        "trust_level": "TRUSTED",
        "crosses_boundary": false
      }
    }
  ],
  "structural_analysis": {
    "critical_paths": [
      {
        "path": ["ACTOR-external_caller", "ACTION-deposit", "ASSET-balance"],

      }

    "state_clusters": [
      {

        "shared_state": ["STATE-balance"],

      "orphan_actions": [],
      "orphan_assets": [],
      "disconnected_requirements": []

```

### Step 4b: Write the File

Write the JSON to `.speca/graph.json` using the Write tool. Use 2-space indentation for readability.

## Integration with Downstream Commands

When `.speca/graph.json` exists, downstream commands can optionally leverage it:

- **`/speca-map`**: Use `critical_paths` and `state_clusters` to prioritize mapping candidates. Functions that appear on critical paths should receive higher confidence scores when mapped.
- **`/speca-checklist`**: Use `state_clusters` to generate additional reentrancy and state-ordering check items. Use `orphans` to flag spec gaps.
- **`/speca-audit`**: Use `critical_paths` to prioritize audit order within the same priority level — items on critical paths are audited first.

These integrations are advisory. Downstream commands MUST function correctly even if `graph.json` does not exist. If `graph.json` exists, read it in Phase 0 and use it to enhance results.

---

## Error Handling

- If `.speca/config.json` does not exist, stop and tell the user to run `/speca-init`.
- If `.speca/requirements.json` does not exist, stop and tell the user to run `/speca-extract`.
- If the requirements array is empty, stop and tell the user to run `/speca-extract`.
- If `.speca/graph.json` already exists, overwrite it without asking (graph generation is idempotent and re-runnable).

---

## Notes

- This command is **non-interactive**. Do not prompt the user for input during graph generation.
- This command is **optional** in the pipeline. The SPECA pipeline functions without it, but graph generation improves downstream precision.
- Entity deduplication is critical. Use semantic similarity (not just string matching) to identify entities that refer to the same concept. For example, "the contract owner", "owner address", and "admin" may all refer to the same actor if the specification treats them equivalently.
- When extracting actions, prefer the most specific verb form. "transfer tokens" is more useful than just "transfer" if the specification distinguishes between ETH transfers and token transfers.
- The graph is a specification-level artifact, not a code-level artifact. It represents what the spec describes, not what the code implements. Discrepancies between the graph and actual code are detected by `/speca-map` and `/speca-audit`.
