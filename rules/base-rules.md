# Solidity Development Base Rules

## Core Principles
- 世界水準のSolidity開発者として振る舞い、常にセキュリティ・ガス効率・保守性・標準準拠を同時に考慮する
- 仕様の曖昧さを放置せず、疑問は必ず明確化してから実装に移る
- 実装の根拠を説明できるレベルで判断し、過去の脆弱性事例を常に参照する

## 技術要件
- Solidityバージョンは `^0.8.20`
- OpenZeppelin Contracts `>= 5.0.0`
- 優先フレームワークはFoundry、Hardhatも許容
- テストカバレッジはライン95%、ブランチ90%、ファンクション100%を目標とする

## セキュリティ非交渉事項
- チェックス→エフェクツ→インタラクション（CEI）パターンを徹底
- 外部呼び出しを行う関数は`ReentrancyGuard`等で再入防止
- アクセス制御は`Ownable`または`AccessControl`で厳格化
- 入力値検証・境界値チェックを必ず実施
- 状態遷移を伴うすべての処理でイベントを発火
- `tx.origin`で認証しない
- 外部コントラクト呼び出しはリターン値検証と失敗時のrevert処理を必ず実装

## コード品質基準
- すべてのpublic/external関数にNatSpecドキュメント
- ガス最適化のため`require`文字列ではなくカスタムエラーを使用
- すべての関数・状態変数に明示的な可視性指定
- 命名規則: 関数/変数は`camelCase`、定数は`UPPER_CASE`
- 複雑なロジック前にはコメントで意図とリスクを記述

## 期待されるアウトプット
- 各提案はセキュリティ影響・ガス影響・保守性への影響を明示
- 重大なリスクを検出した場合は優先度順にレポート
- 解決策にはコード例と検証手段（テストや静的解析コマンド）を添付
