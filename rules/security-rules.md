# Solidity Security Auditor Rules

## 目的
- コードと設計の脆弱性を網羅的に洗い出し、修正策・検証手段を提示する
- 重大度（Critical / High / Medium / Low / Informational）で優先順位付けし、再発防止策まで含めたレポートを作成する

## アセスメントフロー
1. **エントリポイント分析**: external/public関数の権限・状態遷移・資金フローを記録
2. **状態遷移検証**: ステートマシンを文章化し、競合・再入余地を排除
3. **依存関係分析**: 外部コール、オラクル、ライブラリに基づくリスクを特定
4. **自動化ツール**: Slither、Mythril、Foundry invariant/fuzzテストの実行を提案
5. **レポート作成**: 再現手順、影響範囲、推奨修正を明文化

## クリティカルチェックリスト
- 再入（ETH/トークン/クロス関数）
- オーバーフロー/アンダーフロー（`unchecked`使用箇所含む）
- 権限昇格・アクセスポリシー逸脱
- フロントラン（MEV、Pending tx 監視リスク）
- タイム依存（block.timestamp, block.number）
- DOS（ループ、外部呼出し、ガス上限）
- オラクル操作、価格操作
- フラッシュローン依存
- 署名リプレイ、EIP-712検証不備

## 対応パターン
- `nonReentrant` / CEI / Pull Payment
- `AccessControl`ロール設計と多段署名
- `Pausable`による緊急停止
- 署名検証はチェーンID・デッドラインを必須化
- リレイ攻撃防止のためのドメインセパレータ確認

## レポートフォーマット
1. エグゼクティブサマリー
2. 脆弱性一覧（重大度、説明、場所、影響、推奨修正）
3. 再現手順・PoCコード
4. 修正後の検証プラン（テスト、監査ログ）
5. ガス影響とトレードオフ
