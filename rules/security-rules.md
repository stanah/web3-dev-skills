# Solidity Security Auditor Rules

## 目的
- コードと設計の脆弱性を網羅的に洗い出し、修正策・検証手段を提示する
- 重大度（Critical / High / Medium / Low / Informational）で優先順位付けし、再発防止策まで含めたレポートを作成する

## アセスメントフロー
1. **エントリポイント分析**: external/public関数の権限・状態遷移・資金フローを記録
2. **状態遷移検証**: ステートマシンを文章化し、競合・再入余地・ロールバックを排除
3. **依存関係分析**: 外部コール、ブリッジ、オラクル、ライブラリ、L2メッセージングに基づくリスクを特定
4. **自動化ツール**: Slither、Mythril、Foundry invariant/fuzz、Echidna、Manticore等の実行を提案し、CIに組み込む
5. **レポート作成**: 再現手順、影響範囲、推奨修正を明文化

## クリティカルチェックリスト
- 再入（ETH/トークン/クロス関数）
- オーバーフロー/アンダーフロー（`unchecked`使用箇所含む）
- 権限昇格・アクセスポリシー逸脱
- フロントラン・MEV・サンドウィッチ攻撃（ペイロード・オーダリング依存）
- タイム依存（`block.timestamp`/`block.number`/`block.prevrandao`）
- DOS（ループ、外部呼出し、ガス上限、ストレージ書き込み制限）
- オラクル操作、価格操作、データスタレネス
- フラッシュローン依存、攻撃者の資金不要性
- 署名リプレイ、EIP-712/EIP-4337検証不備
- ブリッジ/クロスチェーンメッセージのリプレイ・順序逆転・二重処理
- アップグレード権限のハイジャック、ストレージ衝突、初期化漏れ
- サプライチェーン（依存ライブラリの改竄、Compiler/Reproducible Build問題）

## 対応パターン
- `nonReentrant` / CEI / Pull Payment
- `AccessControl`ロール設計とマルチシグ/Timelock/Guardiansの組み合わせ
- `Pausable`/`CircuitBreaker`/`RateLimiter`による緊急停止・スロットリング
- 署名検証はチェーンID・デッドライン・一意nonce・ドメインセパレータ確認を必須化
- ブリッジ設計では二重確認（ライトクライアント/ガーディアン）、メッセージID追跡、`msg.sender`検証を行う
- イベントと監視ダッシュボードを用意し、Runtime Monitor（Forta等）へフィードする

## レポートフォーマット
1. エグゼクティブサマリー
2. 脆弱性一覧（重大度、説明、場所、影響、推奨修正、EthTrust v3該当カテゴリ）
3. 再現手順・PoCコード
4. 修正後の検証プラン（テスト、監査ログ、Runtime監視）
5. ガス影響・オペレーションリスク・ユーザ影響のトレードオフ
