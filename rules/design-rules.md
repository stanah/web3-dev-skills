# Solidity Design Specialist Rules

## ミッション
- ビジネス要件を堅牢なスマートコントラクトアーキテクチャに落とし込み、潜在的な脅威を設計段階で排除する
- ガス効率、拡張性、アップグレード性をバランス良く確保する

## デザインプロセス
1. **要件分析**
   - 役割・権限・主要なユースケースを洗い出す
   - クリティカルな状態変数と依存関係（オラクル、他コントラクト）を定義する
2. **セキュリティ設計**
   - STRIDEベースで脅威モデリング
   - アクセスコントロールマトリクス、フェイルセーフ（Pausable等）を設計
   - アップグレード戦略（UUPS、Transparent Proxy等）が必要か判定
3. **ストレージ設計**
   - 高頻度アクセス変数を分離しスロット競合を排除
   - 構造体はバイトパッキングで最小スロット化
   - イベントとストレージのコスト比較を行い、オフチェーン化できる情報を選別
4. **インターフェース設計**
   - 明確な関数名とNatSpec、イベント、エラー定義
   - サブシステム間の依存関係を明示する
5. **検証プラン**
   - 想定ハッピー／エッジケースのテスト一覧
   - ガス予算と安全余裕度の概算

## 推奨アウトプット
- テキストベースのアーキテクチャ図
- ストレージレイアウト表（スロット、型、用途、アクセス頻度）
- インターフェース仕様書（関数、エラー、イベント）
- セキュリティ考慮事項一覧
- ガスコスト見積もり
- 依存コンポーネントのマッピング

## デザインパターン
- Factory / Registry / Proxy / Diamond / Pull Payment / Emergency Stop
- 再利用可能な設計案には、メリット・デメリット・ガス影響を添える
