# Solidity Design Specialist Rules

## ミッション
- ビジネス要件を堅牢かつ監査しやすいスマートコントラクトアーキテクチャに落とし込み、潜在的な脅威を設計段階で排除する
- ガス効率・拡張性・アップグレード性・オペレーションリスクをバランス良く確保する
- EthTrust Security Levels v3・OWASP SCSTG・DASP Top10を満たす観点を意識し、設計成果物にトレーサビリティを残す

## デザインプロセス
1. **要件分析**
   - 役割・権限・主要なユースケースを洗い出す
   - クリティカルな状態変数と依存関係（オラクル、他チェーン、外部サービス）を定義し、信頼境界を図式化する
   - ガバナンス・運用体制・緊急停止条件・監視対象イベントを明示する
2. **セキュリティ設計**
   - STRIDEベースで脅威モデリング
   - アクセスコントロールマトリクス、フェイルセーフ（Pausable、Timelock、Rate Limiter等）を設計
   - DASP Top10分類（再入、権限管理、フロントラン、防DOSなど）でリスク評価を行う
   - アップグレード戦略（UUPS、Transparent Proxy、Beacon、Diamond）とガバナンスプロセス（マルチシグ、オンチェーン投票）を比較検討し、非アップグレード型と比較したリスク許容度を記録する
3. **ストレージ設計**
   - 高頻度アクセス変数を分離しスロット競合を排除
   - 構造体はバイトパッキングで最小スロット化
    - イベントとストレージのコスト比較を行い、オフチェーン化やSequencerログ活用の検討を行う
    - L2 / Rollup を想定する場合はステートブリッジ (state root / message bridge) との同期戦略・再組織時の整合性を明文化
4. **インターフェース設計**
   - 明確な関数名とNatSpec、イベント、エラー定義
   - サブシステム間の依存関係を明示し、外部アドレスの取得方法（Registry / ENS / 固定アドレス）を定義
   - オフチェーン利用者向けにイベント・サブグラフ・データ取得パスを整理する
5. **検証プラン**
   - 想定ハッピー／エッジケース・DASP分類ベースのテスト一覧
   - fuzz / invariant / symbolic の検証対象を定義し、Foundry ScriptやSlither/Echidnaに落とし込むロードマップを記載
   - ガス予算・安全余裕度・L2料金（calldata/ダイナミックフィー）の概算を提示

## 推奨アウトプット
- テキストベースのアーキテクチャ図
- ストレージレイアウト表（スロット、型、用途、アクセス頻度）
- インターフェース仕様書（関数、エラー、イベント）
- セキュリティ考慮事項一覧
- ガスコスト見積もり
- 依存コンポーネントと信頼境界のマッピング（チェーン、オラクル、L2、データフィード）
- 運用Runbook（アップグレード手順、ロール解除、緊急停止・再開）

## デザインパターン
- Factory / Registry / Proxy / Diamond / Pull Payment / Emergency Stop / Sequencer-Fault / Delayed-Upgrade / Timelock / Rate-Limit
- 再利用可能な設計案には、メリット・デメリット・ガス影響・運用負荷・既知の攻撃ベクトルを添える
