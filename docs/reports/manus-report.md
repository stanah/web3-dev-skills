# Solidityスマートコントラクト開発特化AIエージェント構築のための調査レポート

## 1. はじめに

本レポートは、Solidityスマートコントラクト開発に特化したAIエージェント（以下、Solidity特化エージェント）の構築を目指すにあたり、その最適化のための**ルールファイル整備**、**ワークフロー整備**、および**言語特化モデルの自作方法**に関する調査結果をまとめたものです。

## 2. 言語特化AIエージェントの事例とアーキテクチャ

他のプログラミング言語に特化したAIエージェントの事例から、Solidity特化エージェントのアーキテクチャ設計に役立つ知見が得られました。

### 2.1. 事例の傾向

*   **マルチエージェントシステム**: 複雑なタスク（例：多言語コード生成、共同開発）に対して、言語ごとに特化したエージェントが**協調**し、知識を共有することでプログラミング能力を高めるアプローチが主流です。
*   **専門性の分離**: エージェントを単一のタスク（例：コード生成、デバッグ、計画検証）に特化させることで、全体としての性能と信頼性を向上させています（例：`CodeSim`）。

### 2.2. Solidity特化エージェントへの応用

Solidity開発の特性（特に**セキュリティの重要性**）を考慮すると、以下のような**マルチエージェントアーキテクチャ**が有効と考えられます。

| エージェント名 | 役割 | 専門知識/ツール |
| :--- | :--- | :--- |
| **Logic Agent** | コントラクトの主要なビジネスロジックを実装 | Solidityの基本文法、ERC標準（ERC-20, ERC-721など） |
| **Security Agent** | セキュリティ脆弱性の検出と修正 | 再入可能性、整数オーバーフロー、アクセス制御、Solidityセキュリティベストプラクティス |
| **Testing Agent** | テストコードの生成と実行 | Foundry/Hardhatなどのテストフレームワーク、単体テスト/統合テストのパターン |
| **Refactoring Agent** | コードの最適化とスタイルガイドの適用 | Solidityスタイルガイド、ガス効率の最適化 |

## 3. ルールファイルとワークフローの設計手法

エージェントの振る舞いを最適化するためには、明確なルールファイルと効率的なワークフローの設計が不可欠です。

### 3.1. ルールファイル（プロンプト）の整備

エージェントの行動を規定するルールは、以下の階層構造で整備することが推奨されます。

| ルール種別 | 適用範囲 | 内容の例（Solidity特化） |
| :--- | :--- | :--- |
| **グローバルルール** | すべてのタスクに共通 | 「あなたは世界トップクラスのSolidity開発者であり、セキュリティ監査人です。」、「回答は常にMarkdown形式で構造化すること。」 |
| **プロジェクトルール** | 特定のプロジェクトに共通 | 「Solidityのバージョンは`0.8.20`を使用すること。」、「ERC-721標準に準拠すること。」、「特定のライブラリ（例：OpenZeppelin）のみを使用すること。」 |
| **タスクルール** | 個別のタスクに特化 | 「この関数は`onlyOwner`修飾子でアクセスを制限すること。」、「再入可能性攻撃を防ぐためにChecks-Effects-Interactionsパターンを適用すること。」 |

これらのルールは、エージェントフレームワーク（例：LangChain, LangGraph）の**システムプロンプト**や**ガードレール**として機能します。

### 3.2. ワークフロー（オーケストレーション）の設計

複雑なSolidity開発タスクは、**LangGraph**などのステートフルなフレームワークを用いて、マルチエージェントワークフローとして設計することが効果的です。

1.  **Plan (計画)**: Logic Agentが要件を分析し、開発ステップを計画。
2.  **Code (実装)**: Logic Agentがコードを生成。
3.  **Review (レビュー)**: Security Agentが生成されたコードをセキュリティ観点からレビューし、脆弱性を指摘。
4.  **Test (テスト)**: Testing Agentが単体テストを生成・実行し、バグを検出。
5.  **Refine (洗練)**: Logic AgentがReview/Testの結果に基づきコードを修正・最適化。
6.  **Complete (完了)**: すべてのテストとセキュリティチェックが合格した場合に完了。

このワークフローにより、エージェントは自律的に**計画→実行→検証→修正**のサイクルを回すことが可能になります。

## 4. 言語特化モデルとその自作方法

Solidity特化エージェントの性能を最大化するためには、基盤となるLLMをSolidityに特化させることが有効です。

### 4.1. 特化モデルの必要性

汎用LLMは広範な知識を持ちますが、Solidity特有の**セキュリティ脆弱性**や**ガス効率**に関する深い理解、および最新の**EVM（Ethereum Virtual Machine）の挙動**に関する知識が不足しがちです。

### 4.2. 自作（ファインチューニング）の方法

既存のコード生成LLM（例：Codestral, Code Llama）をベースに、Solidityに特化したデータで**ファインチューニング**（微調整）を行うのが最も現実的なアプローチです。

| ステップ | 内容 | Solidity特化のための考慮事項 |
| :--- | :--- | :--- |
| **1. データセットの準備** | **高品質**なSolidityコードと、それに対応する**指示**（プロンプト）のペアを収集。 | **セキュリティ監査済みのコントラクト**、**脆弱性とその修正パッチ**、**ガス最適化前後のコード**、**ERC標準の実装例**など、専門性の高いデータを含める。 |
| **2. モデルの選択** | 既存のコード生成に強いLLMを選択（例：MistralのCodestral、MetaのCode Llamaなど）。 | パラメータサイズと利用可能な計算リソースを考慮。 |
| **3. ファインチューニング** | 収集したデータセットを用いて、モデルの重みを更新。**LoRA**（Low-Rank Adaptation）などの効率的な手法が一般的。 | **Instruction Tuning**（指示応答能力の向上）と**Domain Adaptation**（Solidity特有の知識の埋め込み）を目的とする。 |
| **4. 評価** | HumanEvalやMBPPなどのコード生成ベンチマークに加え、Solidity特有の**セキュリティベンチマーク**（例：Slitherのテストケース）で評価。 | **セキュリティ脆弱性の検出・修正能力**を最重要視して評価する。 |

## 5. Solidity特化エージェント設計のベストプラクティス

Solidity特化エージェントのルールファイルとワークフローに組み込むべき、具体的なベストプラクティスをまとめます。

### 5.1. セキュリティに関するルール（最優先事項）

Solidity開発においてセキュリティは最も重要です。エージェントは以下のルールを**常に**適用する必要があります。

*   **Checks-Effects-Interactionsパターン**: 外部呼び出し（Interactions）の前に、すべての前提条件（Checks）と状態変更（Effects）を完了させる。
*   **再入可能性の防止**: 外部コントラクトへの呼び出しには、`call`の代わりに`transfer`や`send`（非推奨だが単純なケースで安全）、または**リエンシーガード**を使用する。
*   **整数オーバーフロー/アンダーフローの防止**: Solidity 0.8.0以降ではデフォルトでチェックされるが、古いバージョンやカスタムロジックでは`SafeMath`ライブラリの概念を適用する。
*   **アクセス制御**: `onlyOwner`や`require`文を用いて、重要な関数へのアクセスを適切に制限する。
*   **イベントの発行**: 重要な状態変更（例：送金、所有権変更）の際には、必ず`event`を発行する。

### 5.2. テストと検証に関するルール

*   **テスト駆動開発（TDD）**: まずテストケースを生成し、それに合格するコードを生成するワークフローを推奨。
*   **テストフレームワークの活用**: **Foundry**（Solidityでテストを記述可能）や**Hardhat**（JavaScript/TypeScriptでテストを記述）の利用を前提とする。
*   **カバレッジの確保**: 生成されたテストコードが、コントラクトのすべてのロジックをカバーしていることを確認する。

### 5.3. スタイルと設計に関するルール

*   **Solidityスタイルガイドの遵守**: 変数名、関数名、インデントなど、公式のスタイルガイドに準拠させる。
*   **コントラクトのサイズ制限**: EVMの制限（24KB）を意識し、コントラクトをモジュール化する。
*   **ガスの最適化**: ループ処理の削減、ストレージ変数の適切なパッキングなど、ガス効率を考慮したコードを生成する。

## 6. 結論

Solidity特化AIエージェントの構築は、単なるコード生成を超え、**セキュリティ監査人の役割**を果たすことが成功の鍵となります。

1.  **アーキテクチャ**: **Security Agent**を含むマルチエージェントシステムを採用する。
2.  **ルール**: セキュリティベストプラクティスを**グローバルルール**として最優先で組み込む。
3.  **モデル**: **セキュリティ脆弱性データ**を含む高品質なデータセットで既存のコードLLMを**ファインチューニング**し、Solidity特化モデルを作成する。

これらの知見に基づき、具体的なエージェントのプロンプトやワークフロー設計に進むことを推奨します。
